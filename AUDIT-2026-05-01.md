## What's already fixed and pushed today

### Commit `e47956f` — CIS write-through gap (architectural)

**Root cause found**: `POST /api/workspace-data` was upserting to `collection_workspace_data` and never calling `recordDecision`. Result: any collection that lived in the Creative block (vibe / consumer / brand-dna / moodboard / synthesis / trends) wrote to workspace storage but never populated the CIS. `compilePromptContext()` (which has no fallback to workspace data, unlike `loadFullContext`) then rendered empty placeholders for `editorial_prompt`, `still_life_prompt`, `copy_prompt`, `seo_prompt`.

**Why it mattered**: SLAIZ had 1 decision in CIS (29 in AZUR). Anything published from SLAIZ via the marketing-prompts.ts templates would have leaked empty placeholders to the user. Production blocker.

**What shipped**:
- New `src/lib/ai/workspace-to-cis.ts` mapper that inverts what `load-full-context.ts` already reads.
- `POST /api/workspace-data` now `await`s `recordDecisions` after the upsert (not fire-and-forget — Vercel kills dangling promises).
- `scripts/backfill-cis-from-workspace.ts` one-shot for legacy collections.
- 3 self-corrections from Codex review applied: `moodboard_analysis` stored as string (was object, mismatched the consumer's typing), arrays sorted for idempotent versioning under user reordering, await replaces fire-and-forget.

**Verified after fix**: SLAIZ 1 → 14 decisions. AZUR intact (29). Re-running the backfill with the bug fixes regenerated only the affected keys (idempotency confirmed).

### Commit `cd55fbe` — Sales Dashboard revenue P0 bug

**Found by walking the SLAIZ Marketing > Sales Dashboard with Playwright at the end of the visual pass.** Dashboard showed €9.2M revenue for SS27 SLAIZ. Math should yield €297K (matches scenario 2 target, matches Collection Builder header).

**Root cause**: `SalesDashboardCard.tsx` lines 39 / 88 / 116 / 140 multiplied `expected_sales × pvp`, but `expected_sales` is already stored as revenue in euros (computed at SKU write time as `buy_units × sale_pct/100 × pvp`). Multiplying by pvp again inflated by a factor of ~30 (the average price).

**Same root cause** as the bug fixed in `load-presentation-data.ts` on 2026-04-30 — the fix did not propagate to the Sales Dashboard component. Identified-by-symptom this time, not by reading code.

**Verified after fix**: dashboard reloaded shows €297K, story split sums exactly (€36K + €154K + €83K + €24K = €297K). Other consumers of `expected_sales` (`GoToMarketCard`, `PaidGrowthCard`, presentation-loader) already read it correctly without multiplication.

### MEDIUM P2 — accepted

- i18n check script (`scripts/check-i18n.sh`) is grep-based and tolerates ±5 diff. There ARE hardcoded strings in `creative/page.tsx`, `CollectionBuilder.tsx`, `TechPackExportSheet.tsx`. Not blocking, but locales are not 100% complete. M effort to fix properly.

### BLIND SPOTS Codex flagged

- **No automated test of the 96 prompt placeholders.** `renderPrompt()` silently swallows missing values. Risk: a Marketing prompt missing `brand_voice_personality` because nobody filled it ships empty to the LLM and the user gets a generic answer. S fix: fixture-based render test.
- **AI endpoints surface raw provider 500s to the user** (`src/lib/ai/llm-client.ts:91-100`). UX risk: customer sees "Anthropic: rate limit exceeded" instead of a friendly message. S fix.
- **No rate limit per user on expensive AI routes** (e.g. `brief/scenarios` does 2 Perplexity + Haiku + Gemini retry). M fix.
- **`collection-assets` storage bucket is deliberately public**. If you ever promise "your designs are private" in marketing, this contradicts it. L fix (signed URLs or private bucket).

## Recommendation

**Launch today is OK** for the headline customer journey. The visible product is gold standard. The math is right after today's Sales Dashboard fix. The CIS pipeline is now coherent end-to-end after the workspace-to-cis hook.

**Three things I'd do before opening the floodgates**:
1. Apply the partial unique index migration (10 min, removes a latent race).
2. Decide and either ship the brand-names scrubber or write a known-issue note explicitly accepting the risk.
3. Decide on SSRF allowlist — either ship today (S effort) or document as known security risk to remediate week 1.

The other P0/P1 items are real but their impact is bounded: imagery quota race only matters once you have concurrent paying users hammering the same endpoint, tech-pack team-aware only matters once a customer invites collaborators. Those can be week-1 work.

The product is much more launch-ready than the previous handoff suggested. The biggest things in this session were not the new fixes — they were verifying that the fixes from prior sessions are still working, and catching the one regression nobody noticed (€9.2M Sales Dashboard).
