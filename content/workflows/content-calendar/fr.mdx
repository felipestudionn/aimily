## Qu'est-ce qu'un content calendar — et pourquoi il appartient dans la même plateforme ?

Un content calendar en mode est la séquence de lancement pour une collection : quel SKU apparaît où, dans quel format, à quel jour. La plupart des marques le gèrent dans un tableur, avec hero assets dans Dropbox, captions dans Notion, et un drop calendar dans Google Calendar. Rien ne lit rien.

Le content calendar d'aimily lit directement le range plan signé. Chaque SKU est déjà une carte. Voix, palette et set de références de l'ADN de marque pré-chargés. L'équipe marketing démarre où l'équipe design a fini — pas depuis un brief PDF qui résume tout en 8 bullets et perd la nuance.

C'est le bloc qui transforme « on a shippé une collection » en « on a shippé une collection sold-out ».

## Comment aimily la gère

Le flow en sept étapes est documenté dans le HowTo schema ci-dessus. La sortie clé :

- **Éditorial AI** on-model par SKU, sans shooting.
- **Drop calendar** verrouillé entre stockists avec dépendances cross-block (stage du tech pack gate la date du drop).
- **Content studio** par SKU par canal — chaque format visuel en un clic.
- **Captions dans la voix de marque**, rotation de hashtags (5 par caption, 3 fixes marque), registre par canal.
- **Sales dashboard** live : revenus jour 1, sell-through, top SKU, gross margin, channel split.

Le stack de modèles : Fal.ai pour la génération de video reel (Kling 2.1 sur les plans Pro), gpt-image-1 pour les éditoriaux, Freepik pour stills + try-on, Anthropic Claude pour la génération de captions + content briefs. Le tout enveloppé dans des clauses de protection IP qui garantissent l'usage commercial.

## Ce que tu obtiens

- **Assets éditoriaux par SKU** — hero on-model, detail close-ups, lifestyle in-context, try-on.
- **Drop calendar** — Gantt avec dépendances stockist + production trackées.
- **Wholesale order sheet** — assortment par buyer, allocations drop-by-drop, status PO → livraison.
- **Content calendar PDF** — plan campagne multi-canal, pipeline visuel à 4 tiers par SKU.
- **Captions par canal + SKU** dans la voix de marque, avec rotation de hashtags.
- **Email sequence** — launch, post-launch, restock, drop reminder. Subject + preheader + body.
- **Sales dashboard** live — revenus, sell-through, top SKUs, KPI canal.
- **Variantes d'asset** pour paid media — chaque aspect ratio, prêt pour le media partner.

## Avant aimily, vs avec aimily

**Avant aimily** — Marketing s'improvise deux semaines avant le lancement. Le photographe annule. Le lookbook part bâclé. La date du drop glisse. Les pré-commandes meurent dans le pipe. L'équipe contenu travaille depuis un brief PDF qui est déjà 4 versions obsolète.

**Avec aimily** — Le jour où un SKU est signé dans le range plan, son éditorial est prêt dans le workspace marketing. Drop calendar verrouillé entre stockists. KPI live avant que l'email de lancement ne sorte.

## Pourquoi ça compte

Le logiciel de mode se sépare exactement à ce point. Les PLM (Centric, PTC FlexPLM) s'arrêtent au tech pack. Les ERP (NetSuite) commencent à la commande. Les outils marketing (Hootsuite, Canva, Klaviyo) commencent à « donne-nous tes assets ». Rien ne connecte le SKU à la campagne.

aimily est la seule plateforme qui boucle la boucle. L'équipe contenu lit les mêmes SKUs que l'équipe design a shippés. Le drop calendar enforce les stage gates du tech pack. Les captions parlent dans la voix d'ADN de marque verrouillée au Bloc 1. Le sales dashboard rattache aux unités forecast dans le range plan.

C'est le moat. Chaque autre plateforme gère un bloc. aimily gère les quatre, avec le flow de données forward.
