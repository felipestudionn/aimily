> Sister to [`shopify-lane-feasibility-2026-05-19.md`](shopify-lane-feasibility-2026-05-19.md). That doc says "what should we build"; this doc says "what we have running today, verified end-to-end".

## §1 · Why this harness exists

The Shopify lane (`/src/lib/strategy/parsers/shopify-graphql.ts` + `/api/strategy/sources/shopify-ingest`) needs a target real Shopify store to verify:
- GraphQL Admin API integration against real schema
- Query cost budget (≤1000 pts per request)
- Velocity windowing over real `processedAt` timestamps
- Refund detection via `order.returns` connection
- End-to-end UI render of verdicts on Shopify data

Building against the legacy CSV/XLSX path (`seed-shopify-real-apparel.ts`) is faster but **does not exercise the live API**. The harness fills that gap.

## §3 · API scopes (17 granted)

Listed in the order they're enabled in the custom-app config:

**Read (13)** — minimum set for the GraphQL adapter:
- `read_analytics`, `read_customers`, `read_discounts`, `read_fulfillments`, `read_inventory`, `read_locations`, `read_orders`, `read_price_rules`, `read_products`, `read_publications`, `read_reports`, `read_returns`, `read_markets`

**Write (4)** — for seeding:
- `write_orders` — `orderCreate`, `refundCreate`
- `write_customers` — `customer.toUpsert` inside orderCreate
- `write_products` — `productSet`, `productDelete`
- `write_inventory` — `inventoryQuantities` in `productSet.variants`

**NOT granted** (intentionally — not needed for current scope):
- `read_all_orders` (gated behind app review; default cap = last 60d is enough — our parser only reads last 21d anyway)
- `read_inventory_item_unit_costs` — turns out `inventoryItem.unitCost` is readable with `read_inventory` for custom apps on dev stores; reserved for prod app submission

## §5 · Scripts

Two scripts in `scripts/` reproduce the harness from scratch:

### `setup-shopify-apparel-store.mjs`

One-shot setup. Idempotent: deletes existing products, then imports the 25 apparel products via `productSet` mutation.

```bash
SHOPIFY_DEMO_SHOP=aimily-mlyel0nm.myshopify.com \
SHOPIFY_DEMO_TOKEN=shpat_xxx \
node scripts/setup-shopify-apparel-store.mjs
```

Flags:
- `--dry-run` — print the first 3 product inputs without firing the mutation
- `--no-delete` — skip Phase 1 (don't wipe existing products)

Expects `/tmp/apparel-plan.json` (generated by the inline Python parser in §7).

### `seed-shopify-live-orders.mjs`

Two phases on a single run:
1. **Phase 3** — create 150 orders, power-law SKU distribution, processedAt distributed over last 14d. Rate-limited to 5/min (Shopify hard cap on dev/trial stores).
2. **Phase 4** — sample 20% of seed orders, fire `refundCreate` with `restockType: RETURN`.

```bash
SHOPIFY_DEMO_SHOP=aimily-mlyel0nm.myshopify.com \
SHOPIFY_DEMO_TOKEN=shpat_xxx \
node scripts/seed-shopify-live-orders.mjs
```

Flags:
- `--refunds-only` — skip Phase 3, run only Phase 4 (useful for retrying refunds after a bug fix without re-seeding 32 min of orders)

Orders are tagged `aimily-seed-2026-05-19` for cleanup or re-targeting.

## §7 · Bugs found & fixed during construction

Logged for future maintainers. Each bug below caused a real failure during this session — the fix line is the final landed state.

### Setup script

| # | Bug | Symptom | Fix |
|---|---|---|---|
| 1 | `productDelete` + `productSet` need `write_products` | `Access denied for productSet field` | Added `write_products` scope |
| 2 | `inventoryQuantities` in productSet needs `write_inventory` | implicit (silent partial fail) | Added `write_inventory` |
| 3 | `useTitleDefault` shortcut dropped `productOptions` | `Product options input is required when updating variants` | Always send `productOptions`, even for single-Title-variant products |
| 4 | Ghost image-only rows in apparel.csv (no SKU, no option, no price) | `Option value does not exist` on subsequent variants | Filter rows with no SKU/option/price in the Python parser |
| 5 | Apostrophe-prefixed SKUs (`'4216` → Excel quirk) | invalid SKU character | Strip leading `'` |

### Order seeder

| # | Bug | Symptom | Fix |
|---|---|---|---|
| 6 | `orderCreate` needs `write_orders` + customer.toUpsert needs `write_customers` | `Access denied for orderCreate field` | Added both scopes |
| 7 | `processedAt` time-of-day randomization could fall in future | `processed_at date cannot be in the future` | Switched to ms-offset-from-now with 60s safety floor |
| 8 | $0 price variants (`FIELDREPORT2`) | `Transactions Amount must be greater than zero` | Accepted as known fail (1/150); could filter $0 variants from plan |
| 9 | `refundLineItems[].restockType: RETURN` needs `locationId` | `You need to set a location to restock items` | Added `locationId` |
| 10 | Bogus-gateway parent transactions lack a `reference` field | `Reference can't be blank` | Omit `transactions` from RefundInput — Shopify auto-computes |

### GraphQL adapter (`shopify-graphql.ts`)

| # | Bug | Symptom | Fix |
|---|---|---|---|
| 11 | Query cost 1752 > 1000 pt cap | `Query cost exceeds limit (1000)` | Reduced nested `first:N` (products 50→20, variants 100→30, inventoryLevels 50→10, orders 100→25, lineItems 100→30, returns 10→5, returnLineItems 50→10) |
| 12 | `LineItem.refundedQuantity` removed from 2026-01 API | `Field doesn't exist on type 'LineItem'` | Removed field — parser never used it (returns come from `order.returns`) |

## §8 · Hard limits we cannot avoid

| Limit | Source | Impact |
|---|---|---|
| 5 orders/min | Shopify dev/trial store policy | 150 orders = 32 min seed time |
| `refundCreate` has no `processedAt` field | API design | Refunds always timestamp "now"; backdating is impossible |
| Single query cost ≤ 1000 pts | Shopify rate-limit policy | Must keep `first:N` numbers low (see fix #11) |
| Orders > 60d need `read_all_orders` scope | Shopify privacy gate | Our parser only reads 21d back, so n/a for now |
| Bogus-gateway transactions don't carry a `reference` | Gateway design | RefundInput must omit `transactions` array |

## §10 · Known gaps + enrichment options (next-phase decision)

What the harness is **missing** compared to what a real DTC demo deserves:

### Gap A — per-variant images
All variants under a Handle currently share the product master image. The CSV has `Variant Image` URLs populated for color-specific shots that aren't being wired up.

**Fix**: in `setup-shopify-apparel-store.mjs`, set `variants[i].file: { originalSource: variant_image }` when populated. Effort: ~30 LOC + re-run setup.

### Gap B — single location
DTC topology = 1 location → distribution verbs (`AMPLIFY_DISTRIBUTION`) degrade. Synthetic floor kicks in.

**Fix**: add 3-5 fake locations (e.g. Madrid, Paris, NYC, LA) via `locationAdd` mutation, then split inventory across them. Effort: ~50 LOC + needs `write_locations` scope.

### Gap C — product cardinality (only 25)
Real Zara-like demos want 100+ products to feel rich.

**Fix**: import additional CSVs from the shopifypartners repo (`jewelry.csv`, `home_decor.csv`, `home_lifestyle.csv`) OR pull from Shopify Tutorial Theme dataset. Effort: ~10 min per CSV (parse + run setup script).

### Gap D — no historical discounts / markdowns
No discount codes or `compareAtPrice` history. Markdown verbs (`MARKDOWN_ACCELERATE`) fire based on stock + velocity heuristics but lack historical price evidence.

**Fix**: use `priceRuleCreate` + `discountCodeBasicCreate` to create historical discount records aligned to specific SKUs. Effort: medium (~100 LOC).

### Gap E — Refund timestamps all "now"
Refunds bunch on today, distorting the temporal returns curve. Inherent Shopify limitation.

**Mitigation**: accept it, or supplement by creating Loop/Returnly-shaped synthetic data via metafields. Effort: open question.

### Gap F — Order history capped at 14 days back
Our adapter only reads 21d, but the demo could show more if we extend the seeder window.

**Fix**: extend `buildTimestamps` to cover 60d, requires also raising the adapter's `ordersFromDate` from `anchor - 21` to `anchor - 60`. Effort: trivial (2 lines).

## §11 · "What freedom do we have?"

**Full read+write on**: products, variants, inventory levels, orders, refunds, customers, files (per-variant or per-product images), tags, metafields, product types, vendor, prices, compareAtPrice, COGS via inventoryItem.cost, transactions (bogus gateway), discounts (via add'l scope), price_rules (via add'l scope).

**Cannot modify** (without extra scopes or platform features):
- Multiple stores (need different shop_domain + Plus features)
- Online storefront theme programmatically (need Shopify Theme API + write_themes)
- `read_all_orders` (need app submission to Shopify Partner program)
- Shopify Flow workflows (Plus feature, separate API)
- Markets pricing per region (need Markets API)
- Real payment processing (only Bogus gateway in dev stores)

**Practical conclusion**: we can build a demo with realistic apparel-DTC topology — multi-product, multi-color, multi-location, with historical discounts and rich imagery — entirely via the Admin API. The only thing we cannot fake well is **multi-store-fleet distribution** (a 100-store Zara-like setup) because Shopify dev stores are scoped to one shop. But we CAN simulate 5-10 locations within one shop, which is closer to Mango's footprint than Zara's anyway.
